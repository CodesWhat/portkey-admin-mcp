# portkey-admin-mcp — Multi-Agent Audit (June 2026)

> Generated by a multi-agent audit workflow (12 agents): six dimension auditors → adversarial verification → synthesis. Triggered by the Palo Alto Networks acquisition of Portkey (completed 2026-05-29).

**Recommendation: `maintenance-mode`** (confidence: high)

> API is live and working today — but post-PANW acquisition uncertainty warrants a holding pattern, not a burial.

## Executive summary

portkey-admin-mcp v0.3.2 is a well-engineered MCP server with ~150 tools, strong protocol compliance, a passing 118-test suite, and a live API that returned valid JSON as of June 2026. The PANW/Prisma AIRS acquisition closed 2026-05-29 with zero announced API changes, and analyst consensus gives a 12–18 month value-drift window before migration pressure becomes hard to ignore.

The project has had no commits since that acquisition date. There are two confirmed high-severity security gaps (DNS-rebinding via unvalidated Host header; partial SSRF in PORTKEY_BASE_URL validation), five exploitable transitive CVEs fixable by `npm audit fix`, and no user-facing acknowledgment of the acquisition or the resulting uncertainty.

The right call today is **maintenance mode** — patch the security gaps and CVEs, add an acquisition notice, then freeze feature work until PANW publishes a post-integration roadmap (expected ~Q3 2026). Do not archive yet: the API is live, the tooling works, and destroying something functional before a hard deadline is wasteful.

Portkey's March 2026 open-sourcing of its gateway (including SaaS-only governance features) means a self-hosted migration path already exists; the project's `PORTKEY_BASE_URL` env var already supports switching to it.

**Development activity.** The last commit is dated before the acquisition (zero commits since 2026-05-29). The most recent functional release is v0.3.2 (April 2026); subsequent commits were docs-only.

## Prioritized actions

1. **Run `npm audit fix`**. Resolves all five CVEs (fast-uri HIGH ×2, hono moderate ×7, qs moderate, ip-address moderate) in a single lockfile-only operation. Zero breaking changes.

2. **Add acquisition notice to `README.md`**. A callout near the top stating the 2026-05-29 PANW acquisition, that the API surface may change under Prisma AIRS ownership, and that the project is in maintenance mode. Confirmed high-severity doc gap.

3. **Wire `isAllowedHost` into HTTP middleware** (`src/lib/http-app.ts`, `src/lib/security.ts`). The function already exists; call it on `req.headers.host` in a `hostValidationMiddleware` inserted before `mcpAuthMiddleware`. Closes the confirmed DNS-rebinding gap for the `none`-auth deployment mode.

4. **Extend `validateUrl` in `src/services/base.service.ts`** to reject loopback (127.0.0.0/8, ::1), link-local (169.254.0.0/16), and RFC-1918 destinations, or add a `*.portkey.ai` hostname allowlist. Closes the partial SSRF vector.

5. **Fix `run_` and `test_` prefixes in `src/tools/index.ts:101`**. Remove from `READ_ONLY_NON_IDEMPOTENT_TOOL_PREFIXES`. Both `run_prompt_completion` (billable LLM call) and `test_mcp_server` (external connectivity probe) are falsely marked `readOnlyHint: true`.

6. **Fix HTTP 400 → 404 for session-not-found** in `src/lib/http-app.ts` lines 725, 830, 892. Three-line change; the SDK's own JSDoc confirms 404.

7. **Re-record contract fixtures** via `npm run record:fixtures` and add a `manifest.json` with a `recorded_at` timestamp. Current fixtures are pre-acquisition.

8. **Add `--experimental-test-coverage`** to the `test` script in `package.json`. Built into Node 20+.

9. **Add `needs: [ci]` to `release.yml`** to prevent publishing a broken build on a tagged commit.

10. **Align `engines` field to `>=24`** in `package.json` (or lower esbuild target to `node20`).

## Recommendation

**Maintenance mode** — not archive-now.

### Rationale

The API-currency dimension is the decisive input, and its evidence is unambiguous: the API returned a valid live response on June 5, 2026, the status page shows 99.978% uptime, and no sunset dates or breaking-change notices exist. The project works end-to-end right now.

Archiving immediately would destroy a functional tool before any deadline exists. The 12–18 month analyst drift estimate is an *uncertain* industry-pattern extrapolation, not a statement from PANW or Portkey — it should inform planning horizon, not trigger a hard shutdown.

What the evidence does support is **stopping active feature development** and entering a minimal-maintenance posture:

- Patch the two confirmed high-severity security gaps (DNS-rebinding, partial SSRF) and the CVEs. Maintenance-level fixes, not new features.
- Add the acquisition notice to the README so new users can make an informed decision.
- Watch for PANW's post-integration roadmap announcement, expected Q3 2026. That announcement — not analyst speculation — is the actual trigger for re-evaluating archive-now vs. migrate.
- If PANW announces API deprecation or domain migration, the `PORTKEY_BASE_URL` env var already exists to point the server at a self-hosted Portkey gateway (open-sourced March 2026), making a future migration significantly easier than a full rewrite.

### Concrete next steps

1. Patch and release as v0.3.3 (CVEs + security fixes + acquisition notice) before putting the repo into reduced-activity mode.
2. Set a calendar reminder for Q3 2026 to check PANW/Prisma AIRS roadmap communications.
3. If no official API changes by Q4 2026, reassess based on actual API drift (re-run endpoint spot-checks, re-record fixtures).
4. If PANW announces a migration target, open a `migrate` track pointing to the self-hosted gateway or whatever replacement API surface they publish.
